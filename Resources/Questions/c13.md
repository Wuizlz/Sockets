## TCP and UDP Notes

### 1. When `SimpleServer` runs, does it send `"Hi there"` into its output stream and the client read it from its input stream?
Yes. The server writes `"Hi there"` to the socket's output stream with `writeUTF(...)`. The client reads from the socket's input stream with `readUTF(...)` and then prints the returned string.

### 2. Is there a buffer at all?
Yes. There are send and receive buffers managed mostly by the operating system's networking stack. The program does not directly see them, but socket I/O uses them underneath.

### 3. What exactly is UTF?
UTF means `Unicode Transformation Format`. It is a way to encode characters into bytes so they can be sent through a stream or stored in memory.

In this example, `writeUTF(...)` and `readUTF(...)` use Java's modified UTF-8 format:

- the sender writes the string length first
- then it writes the encoded bytes
- the receiver reads the length
- then it reads exactly that many bytes and rebuilds the `String`

### 4. Is a string 8 bytes or 4 bytes?
Not fixed. A Java `String` does not have one universal byte size.

For the specific string `"Hi there"`:

- it has `8` characters
- in UTF-8, those characters use `8` data bytes because they are plain ASCII
- `writeUTF("Hi there")` sends `10` bytes total because it adds a `2`-byte length prefix before the text bytes

### 5. What is a `DatagramSocket`?
A `DatagramSocket` is a local UDP communication endpoint bound to a port. It can send and receive UDP datagrams directly.

Unlike TCP:

- there is no `ServerSocket`
- there is no `accept()`
- there is no continuous byte stream

### 6. What is a `DatagramPacket`?
A `DatagramPacket` represents one UDP message. It contains:

- a byte array for the data
- the data length
- the address and port information

### 7. Is `DatagramSocket` the tunnel and `DatagramPacket` the data inside it?
That is close, but a better model is:

- `DatagramSocket` = the local mailbox or endpoint
- `DatagramPacket` = one envelope or message placed into or taken out of that mailbox

UDP is packet-based, not stream-based, so it is better to think in terms of separate messages rather than a continuous tunnel of bytes.

### 8. Do the client and server need to be connected before UDP can send data?
Not in the TCP sense. UDP usually does not create a real connection. A client can send a packet directly to a server's IP address and port with no handshake.

`DatagramSocket.connect()` exists, but for UDP it mostly means "restrict this socket to one peer by default." It does not create a TCP-style stream connection.

### 9. What does `socket.receive(packet)` mean?
It means the current thread stops at that line and waits until one UDP datagram arrives.

Once a packet arrives:

- Java copies the datagram bytes into the packet's byte array
- Java updates the packet length
- `receive(...)` returns
- the next lines of code continue running

### 10. Does `socket.receive(packet)` put the received data into a byte array buffer of size `100`?
Yes, if your code looks like this:

```java
byte[] buffer = new byte[100];
DatagramPacket packet = new DatagramPacket(buffer, buffer.length);
socket.receive(packet);
```

then Java copies the incoming datagram into `buffer`, up to `100` bytes.

Important details:

- `packet.getLength()` tells you how many bytes were actually received
- bytes past `packet.getLength()` are not part of the message
- if the datagram is larger than `100` bytes, it may be truncated

### 11. Does `while (true)` mean a packet arrived?
No. `while (true)` only means "keep repeating forever."

In code like this:

```java
while (true) {
    socket.receive(packet);
    // process packet
}
```

the roles are:

- `while (true)` = keep the server alive so it can handle many packets
- `socket.receive(packet)` = wait for one packet to arrive

So `while (true)` does not detect packets. It just lets the server keep calling `receive(...)` again and again.

### 12. What does it mean that `receive(...)` blocks?
It means the thread pauses at that line until a packet arrives.

Example:

```java
System.out.println("before");
socket.receive(packet);
System.out.println("after");
```

If no packet arrives, `"after"` does not print yet. The thread is waiting inside `receive(...)`. As soon as one UDP datagram arrives, `receive(...)` returns and execution continues.
